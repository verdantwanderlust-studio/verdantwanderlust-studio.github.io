# 網站健檢 — check-status

> Ivy 想知道「網站現在到底好不好」。你的工作:查三件事、翻譯成她聽得懂的話、給她一個清楚的綠燈或紅燈結論。**這 skill 只看、不改。**

---

## 何時觸發

Ivy 講話像這樣就走這 skill:

- 「網站還好嗎?」
- 「幫我看一下狀態」
- 「網站怪怪的,是不是壞了?」
- 「我上次改了什麼來著?」
- 「最近這幾天網站有沒有動過?」
- 「我發布次數是不是快用完了?」
- 「這個月還可以改幾次?」

**不要走這 skill 的情況**:
- 「網站壞了,救回來 / 還原」→ `check-status` 先查清楚,接著走 `publish` 的 rollback
- 「幫我改 / 發布 ...」→ 對應的內容 skill 或 `publish`

這 skill 是「先看一下」,不負責修。查完如果有問題,再轉去 `publish` 或找 Jason。

---

## 背景

- **Repo**:`verdantwanderlust-studio/verdantwanderlust-studio.github.io`,正式網站 = `main` 分支
- **部署**:每次 `main` 有改動,Cloudflare Pages 自動重建。Cloudflare 會把「這次建好了沒、成功還失敗」回報到 GitHub,你透過 Connector 讀得到
- **build 額度**:Cloudflare Pages 免費版每月 500 次 build。每 push 一次(不管 main 還是 preview 分支)約算一次 build
- 全部用 GitHub Connector 讀,**這 skill 不做任何 commit / 改動**

---

## 三項檢查

### 檢查 1 — 網站現在是好的嗎(綠燈 / 紅燈)

透過 Connector 看 `main` 最新一次 commit 的部署狀態(Cloudflare Pages 回報的 commit status / deployment status):

- **成功(綠燈)** → 網站是好的
- **失敗(紅燈)** → 最近一次更新沒上去,網站可能停在舊版或出問題
- **還在建(黃燈 / pending)** → 剛剛才 push,還在跑,請她 1-2 分鐘後再看

### 檢查 2 — 最近改了什麼

透過 Connector 看 `main` 最近 5 筆左右 commit,整理成「時間 + 白話描述」。

commit message 如果是英文或技術話術,翻成 Ivy 看得懂的(「更新首頁主標語」而不是 `update index.mdx`)。

### 檢查 3 — 這個月發布次數用了多少

Cloudflare 後台才有精確數字,你讀不到,所以用**估算**:

1. 透過 Connector 數這個月(從 1 號到今天)所有分支的 push / commit 筆數 — main 加上 preview 分支都算
2. 每筆約等於一次 build
3. 跟每月上限 500 次比

這是估計值,不是精準值,回報時要講明。500 次其實非常多 — 正常維護一個月用個位數到二三十次,除非短時間瘋狂改,不然不會接近上限。

## 怎麼把結果講給 Ivy 聽

三項查完,組一段她看得懂的回報。網站正常時範本:

```
幫你看過了,網站一切正常:

· 現在狀態:正常運作中(最近一次更新成功上線)
· 最近改動:
  - 今天 14:37 更新首頁主標語
  - 5/14 新增大安區陽台案例
  - 5/12 調整服務頁字級
· 這個月發布次數:大約用了 8 次,額度 500 次,還很充足

沒什麼要擔心的。
```

網站有狀況時範本:

```
幫你看過了,有一個地方要注意:

· 現在狀態:最近一次更新「更新首頁主標語」沒有成功上線(紅燈)
  → 目前客人看到的應該還是上一個版本
· 這個月發布次數:大約 8 次,額度還很夠,不是這個問題

這種「更新失敗」我沒辦法自己修。要不要我:
1. 把錯誤狀況整理成一段話,你轉給 Jason
2. 或先幫你還原到上一個正常版本(走還原流程)
```

回報原則:

- **先講結論**(正常 / 有狀況),再講細節。Ivy 要的是安心,不是 log
- 數字一定附參照(「8 次 / 共 500 次」),不要只丟一個裸數字
- 額度估算講明是「大約」
- 不要貼原始 commit hash、英文 log、技術錯誤訊息給她看

## 發現問題時怎麼辦

這 skill 只診斷,不修。查到問題依狀況轉介:

| 查到的狀況 | 下一步 |
|---|---|
| 最近一次部署紅燈、但網站還開得起來 | 多半是某次更新沒上。問她要還原(走 `publish` rollback)還是找 Jason |
| 網站整個打不開 / 白屏 | 找 Jason。你幫她把狀況整理成一段話 |
| build 次數異常高(接近上限) | 提醒她最近是不是改很多次。短期內少開幾次 preview。真的接近 500 才需要緊張 |
| 一切正常 | 跟她說沒事,問要不要順便做點什麼 |

「整理狀況給 Jason」的範本:

```
我幫你寫好了,複製這段傳給 Jason 就行:

---
網站最近一次更新失敗。
- Repo:verdantwanderlust-studio/verdantwanderlust-studio.github.io
- 失敗的更新:「更新首頁主標語」,時間 [日期時間]
- 現象:[紅燈 / 網站打不開 / 你觀察到的]
---
```

## 邊界(以下要回「這要找 Jason」)

- **修 build 失敗**:查得到失敗,但修不是這 skill 的事 → 整理狀況給 Jason
- **進 Cloudflare 後台看精確用量 / 部署 log**:你只讀得到 GitHub 那側,後台數字要 Jason
- **改任何東西**:這 skill 純診斷。要改 / 還原 → 走 `publish`
- **效能、SEO、流量分析**:「網站慢不慢」「Google 搜不搜得到」「多少人來看」不在這 skill 範圍,如 Ivy 問,記下來給 Jason

## Gotchas(踩雷紀錄)

> 第一次跑 skill,先留著。之後遇到問題寫進來。

### G-1:把「還在建」當成「壞了」

剛 push 完 1-2 分鐘內,狀態是 pending(黃燈),這是正常的、不是故障。先確認不是「剛剛才改完還在跑」再下結論。

### G-2:build 次數是估算,不要講死

你數的是 GitHub 的 push 筆數,跟 Cloudflare 實際 build 數不一定 1:1(retry、其他觸發都可能差)。一律講「大約」,要精確數字請 Jason 看後台。

### G-3:她其實是想改東西

「最近改了什麼」有時是「我想再改,先看一下上次改到哪」。查完可以順口問:「你是想接著改哪裡嗎?」接得上對應的內容 skill。

### G-4:報喜也報憂,但不要嚇她

網站正常就明確說正常,不要硬找小問題。真的有狀況也平穩講、給她下一步選項,不要用「嚴重」「危險」這種字眼讓她緊張。
